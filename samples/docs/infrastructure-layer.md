# Infrastructure Layer

The Infrastructure layer provides the **concrete implementations** of all abstractions defined in the Application layer. It is the only layer permitted to reference EF Core, database drivers, typed HTTP clients, and messaging SDKs. Nothing above it depends on it directly — it is wired into the host via dependency injection.

## Persistence models

> 🗃️ **Tooling**: the database schema (tables, indexes, outbox infrastructure), persistence model classes (`Persistence/*.g.cs`), and the EF Core `DbContext` partial (`Repositories/*DbContext.g.cs`) are all generated by the domain's `*.Database` project. See [tooling.md](tooling.md#database-migration-database) for details.

The `Persistence/` sub-folder contains **database-model classes**

Each persistence model extends `ModelBase<TId>` (CoreEx) and may implement marker interfaces such as `ILogicallyDeleted` to participate in automatic soft-delete query filtering.

```csharp
// samples/src/Contoso.Products.Infrastructure/Persistence/Product.g.cs  (source-generated)
public partial class Product : ModelBase<string>, ILogicallyDeleted
{
    public string Sku { get; set; } = default!;
    public string SubCategoryCode { get; set; } = default!;
    // ...
}
```

> **See also**: [`ModelBase<TId>`](../../src/CoreEx.EntityFrameworkCore/ModelBase.cs) · [`ILogicallyDeleted`](../../src/CoreEx.EntityFrameworkCore/ILogicallyDeleted.cs)

## Repositories

Repository implementations live in `Repositories/` and implement the `IXxxRepository` interfaces declared in the Application layer. They use CoreEx's `EfDb<TContext>` abstraction, which provides strongly-typed, mapper-aware access patterns (`GetAsync`, `CreateAsync`, `UpdateAsync`, `DeleteAsync`, `Query`) built on top of EF Core, adding CoreEx's `DataResult<T>` return semantics, change-tracking integration, and query filtering support.

```csharp
// samples/src/Contoso.Products.Infrastructure/Repositories/ProductRepository.cs
[ScopedService<IProductRepository>]
public class ProductRepository(ProductsEfDb ef) : IProductRepository
{
    public Task<Contracts.Product?> GetAsync(string id) => _ef.Products.GetAsync(id);
    public Task<DataResult<Contracts.Product>> CreateAsync(Contracts.Product product) => _ef.Products.CreateAsync(product);
    // ...
}
```

The `EfDb` sub-class (`ProductsEfDb` / `ShoppingEfDb`) acts as the **unit-of-work facade** over the DbContext, declaring a typed property for each entity model and configuring options such as logical-delete filters. The `DbContext` itself is intentionally thin — it delegates connection management to CoreEx's `IDatabase` abstraction so the same connection/transaction is shared across a request.

```csharp
// samples/src/Contoso.Products.Infrastructure/Repositories/ProductsEfDb.cs
public sealed class ProductsEfDb(ProductsDbContext dbContext) : EfDb<ProductsDbContext>(dbContext, _options)
{
    public EfDbMappedModel<Contracts.Product, Persistence.Product, ProductMapper> Products
        => Model<Persistence.Product>().ToMappedModel<Contracts.Product, ProductMapper>(ProductMapper.Default);
    // ...
}
```

> **See also**: [`EfDb<TContext>`](../../src/CoreEx.EntityFrameworkCore/EfDb.cs) · [`EfDbModel<T>`](../../src/CoreEx.EntityFrameworkCore/EfDbModel.cs) · [`EfDbMappedModel<TContract,TModel,TMapper>`](../../src/CoreEx.EntityFrameworkCore/EfDbMappedModel.cs)

## Mapping

The `Mapping/` sub-folder contains **bidirectional mappers** that translate between Contract types and Persistence model types. They extend `BiDirectionMapper<TSource, TDest, TSelf>` and are source-generated for reference-data types; hand-authored for entity types that require custom field mapping.

```csharp
// samples/src/Contoso.Products.Infrastructure/Mapping/ProductMapper.cs
public class ProductMapper : BiDirectionMapper<Contracts.Product, Persistence.Product, ProductMapper>
{
    protected override Persistence.Product OnMap(Contracts.Product source) => new() { Sku = source.Sku!, ... };
    protected override Contracts.Product OnMap(Persistence.Product source) => new Contracts.Product { Sku = source.Sku, ... };
}
```

## External clients and adapter implementations

When a domain needs to call another domain's API over HTTP, the raw HTTP interaction is encapsulated in a **typed HTTP client** (`Clients/`) — a thin wrapper around `HttpClient` that serializes requests and maps responses to CoreEx `Result` types. The adapter implementation (`Adapters/`) then uses this client alongside other infrastructure concerns (local EF store, event publishers) to fulfill the Application-layer adapter interface.

```csharp
// samples/src/Contoso.Shopping.Infrastructure/Clients/ProductsHttpClient.cs
public class ProductsHttpClient(HttpClient httpClient)
{
    public async Task<Result> CreateReservationAsync(MovementRequest request)
    {
        var response = await _httpClient.PostAsJsonAsync("api/inventory/reserve", request, JsonDefaults.SerializerOptions);
        return await response.ToResultAsync();
    }
}

// samples/src/Contoso.Shopping.Infrastructure/Adapters/ProductAdapter.cs
[ScopedService<IProductAdapter>]
public class ProductAdapter(ShoppingEfDb ef, IEventPublisher eventPublisher, ProductsHttpClient client, ...) : IProductAdapter
{
    // GetAsync reads from the local replicated store (EF); ReserveInventoryAsync calls the HTTP client in real time.
}
```

This split — client for transport, adapter for orchestration — keeps each class focused on a single responsibility and makes the HTTP interaction independently testable.
