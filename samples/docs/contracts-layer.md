# Contracts Layer

The Contracts layer defines the **public-facing surface** of a domain — the types that travel over the wire as API request/response payloads and as messaging event bodies. Keeping this boundary explicit prevents internal implementation details from leaking to consumers and makes versioning tractable. Contracts are intentionally thin: they carry data and metadata but no business logic.

- [`samples/src/Contoso.Products.Contracts`](../src/Contoso.Products.Contracts)

## Unified API and messaging surface

By convention, the same contract type is used for **both** the HTTP API response and the messaging event payload. A `Product` returned from `GET /api/products/{id}` is the same `Contracts.Product` type published as a `product.created` event body. A consumer does not need a separate DTO regardless of whether they integrate via REST or by subscribing to the event stream.

The rationale is straightforward: an entity is an entity. Duplicating it into separate API and event representations introduces divergence risk, extra mapping code, and cognitive overhead for consumers. Keeping a single contract type as the authoritative representation means any schema change is made once and is immediately consistent across both integration surfaces.

In practice this flows naturally through the Application layer: the service operates on and returns a `Contracts.Product`, and that same value is handed directly to `EventData.CreateEventWith()` when publishing:

```csharp
// samples/src/Contoso.Products.Application/ProductService.cs
var dr = await _repository.CreateAsync(product).ConfigureAwait(false);
return dr.WhereMutated(v => _unitOfWork.Events.Add(EventData.CreateEventWith(v, EventAction.Created)));
```

On the consuming side, the Shopping domain's `ProductModifySubscriber` deserialises the inbound event directly into the same `Product` contract type (re-declared locally in the Application layer as an internal adapter model) and passes it straight through to the adapter:

```csharp
// samples/src/Contoso.Shopping.Subscribe/Subscribers/ProductModifySubscriber.cs
[Subscribe("contoso.products.product.created.v1")]
[Subscribe("contoso.products.product.updated.v1")]
public class ProductModifySubscriber(IProductSyncAdapter adapter) : SubscribedBase<Product>
{
    public override IValidator<Product>? ValueValidator => ProductValidator.Default;

    protected override Task<Result> OnReceiveAsync(Product value, EventData @event, EventSubscriberArgs args, CancellationToken cancellationToken = default)
        => _adapter.ModifyAsync(value);
}
```

> **Note**: Where a domain consumes another domain's events, the receiving domain declares its own internal representation of that type (e.g. `Contoso.Shopping.Application.Adapters.Products.Product`) rather than taking a hard dependency on the publishing domain's Contracts assembly. The shape is kept deliberately consistent with the published contract, but ownership remains with the consuming domain — preserving the anti-corruption boundary.

## Entity contracts

Entity contracts represent the resource types exposed by the API. They are declared as `partial` classes annotated with `[Contract]`, which triggers the CoreEx source generator ([`CoreEx.Generator`](../../gen/CoreEx.Generator)) to emit supporting members (equality, cloning, JSON serialisation helpers, etc.) into a paired `*.g.cs` file, keeping hand-authored code free of boilerplate.

Mutable resources typically compose three cross-cutting interfaces:

| Interface | Purpose |
|---|---|
| `IIdentifier<T>` | Server-assigned primary key (`Id` property). |
| `IETag` | Optimistic-concurrency token (`ETag` property). |
| `IChangeLog` | Audit trail — `CreatedBy`, `CreatedOn`, `UpdatedBy`, `UpdatedOn`. |

Server-managed fields are decorated with `[ReadOnly(true)]` so they are excluded from inbound requests, by the likes of NSwag (Swagger), automatically.

```csharp
// samples/src/Contoso.Products.Contracts/Product.cs
[Contract]
public partial class Product : ProductBase, IETag, IChangeLog
{
    [ReadOnly(true)] public ChangeLog? ChangeLog { get; set; }
    [ReadOnly(true)] public string? ETag { get; set; }
}
```

An abstract base class (e.g. [`ProductBase`](../src/Contoso.Products.Contracts/ProductBase.cs)) is used where multiple projections of the same resource share common fields — for example `Product` (full detail) and `ProductLite` (query list projection).

> **See also**: [`IIdentifier`](../../src/CoreEx/Entities/IIdentifier.cs) · [`IETag`](../../src/CoreEx/Entities/IETag.cs) · [`IChangeLog`](../../src/CoreEx/Entities/IChangeLog.cs)

## Reference data

> 🛠️ **Tooling**: reference-data contract types (`*.g.cs`) are generated by the domain's `*.CodeGen` project from `ref-data.yaml`. See [tooling.md](tooling.md#code-generation-codegen) for details.

Reference data types represent controlled vocabularies

```csharp
// samples/src/Contoso.Products.Contracts/MovementKind.g.cs  (source-generated)
[ReferenceData]
public partial class MovementKind : ReferenceData<MovementKind> { }
```

A hand-authored partial class can extend the generated type when additional computed members are needed:

```csharp
// samples/src/Contoso.Products.Contracts/UnitOfMeasure.cs
public partial class UnitOfMeasure
{
    [JsonIgnore]
    public int Precision => 16 - Scale;
}
```

> **See also**: [`ReferenceData<T>`](../../src/CoreEx.RefData/ReferenceData.cs) · [`[ReferenceData]`](../../src/CoreEx.RefData/ReferenceDataAttribute.cs)

## Source generation

The `[Contract]` and `[ReferenceData]` class annotations drive the [`CoreEx.Generator`](../gen/CoreEx.Generator) Roslyn source generator.

The [`Contoso.Products.CodeGen`](../src/Contoso.Products.CodeGen) and [`Contoso.Shopping.CodeGen`](../src/Contoso.Shopping.CodeGen) projects also emit generated source code for the reference-data contracts. The [`CoreEx.CodeGen`](../src/CoreEx.CodeGen) framework orchestrates the underlying code-generation process; this is manually initiated by the developer.

> **Convention**: files ending in `.g.cs` are machine-authored. Files without the `.g` suffix are hand-authored and may extend the generated partial with additional members or attributes.
